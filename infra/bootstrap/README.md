# Terraform State Bootstrap

This Terraform configuration creates the S3 bucket used to store remote Terraform state for the AWS Voting Platform project.

## Purpose

Terraform stores information about managed infrastructure in a state file.

The state file can contain:

- AWS resource identifiers;
- resource attributes;
- relationships between resources;
- outputs generated by Terraform;
- potentially sensitive infrastructure information.

For that reason, the main project state should not be stored only on a developer's computer or committed to Git.

This bootstrap configuration creates a secure S3 bucket that will later be used as the remote backend for the development environment.

## Created resources

The configuration creates:

- an Amazon S3 bucket for Terraform state;
- S3 versioning;
- server-side encryption using SSE-S3;
- full S3 Block Public Access;
- bucket owner enforced object ownership;
- a bucket policy denying non-TLS requests.

The bucket resource also uses Terraform `prevent_destroy` protection.

## Authentication

Terraform uses credentials from the standard AWS credential chain.

For local development, activate the AWS CLI profile:

```bash
export AWS_PROFILE=aws-voting-platform
export AWS_REGION=eu-central-1
export AWS_DEFAULT_REGION=eu-central-1
```

Verify the active AWS identity:

```bash
aws sts get-caller-identity
```

No AWS access keys or secret keys are stored in the Terraform configuration.

## Initialization

Enter the bootstrap directory:

```bash
cd infra/bootstrap
```

Initialize Terraform:

```bash
terraform init
```

This downloads the required AWS provider and creates the local `.terraform` working directory.

## Formatting and validation

Format the Terraform files:

```bash
terraform fmt -recursive
```

Check formatting:

```bash
terraform fmt -check -recursive
```

Validate the configuration:

```bash
terraform validate
```

## Planning

Create a Terraform execution plan:

```bash
terraform plan -out=tfplan
```

Inspect the saved plan:

```bash
terraform show tfplan
```

The plan should only create the bootstrap resources. It should not modify or destroy any existing infrastructure.

## Applying

Apply the reviewed plan:

```bash
terraform apply tfplan
```

After a successful apply, display the outputs:

```bash
terraform output
```

Retrieve only the state bucket name:

```bash
terraform output -raw state_bucket_name
```

## AWS verification

Store the bucket name in a shell variable:

```bash
STATE_BUCKET="$(terraform output -raw state_bucket_name)"
```

Check S3 versioning:

```bash
aws s3api get-bucket-versioning \
  --bucket "$STATE_BUCKET"
```

Check server-side encryption:

```bash
aws s3api get-bucket-encryption \
  --bucket "$STATE_BUCKET"
```

Check public access blocking:

```bash
aws s3api get-public-access-block \
  --bucket "$STATE_BUCKET"
```

Check ownership controls:

```bash
aws s3api get-bucket-ownership-controls \
  --bucket "$STATE_BUCKET"
```

## State protection

The S3 bucket uses versioning so previous versions of the Terraform state can be recovered.

The bucket is protected by:

```hcl
lifecycle {
  prevent_destroy = true
}
```

Terraform will therefore reject a plan that attempts to destroy the bucket.

This protection should not be removed without a deliberate state migration or decommissioning procedure.

## Files that must not be committed

The following files and directories are excluded by `infra/.gitignore`:

- `.terraform/`;
- `*.tfstate`;
- `*.tfstate.*`;
- Terraform plan files;
- local `*.tfvars` files;
- local backend configuration;
- Terraform crash logs.

The provider dependency lock file should be committed:

```text
.terraform.lock.hcl
```

It records the selected provider versions and checksums.

## Important bootstrap limitation

The bootstrap configuration initially uses local Terraform state because the remote S3 backend does not exist yet.

The sequence is:

1. run the bootstrap configuration locally;
2. create the S3 state bucket;
3. create the main environment configuration;
4. configure the main environment to use the S3 backend;
5. enable native S3 state locking with `use_lockfile = true`.

The bootstrap state should be stored securely until a dedicated migration strategy is introduced.
